# punycode.ucs2.decode — Specification

## Purpose
This operation converts an input text string into a sequence of Unicode code points expressed as integers. Because the source environment exposes strings as a sequence of 16-bit code units (UCS-2), two adjacent code units that form a valid UTF-16 surrogate pair must be recombined into a single supplementary-plane code point, while unmatched surrogate code units must be preserved individually as raw integer values. The routine described here is the private `ucs2decode` helper exposed publicly as `punycode.ucs2.decode`, per `punycode.js:101-123` and the public-API wiring at `punycode.js:433-436`.

## Contract
- Input: a text string interpreted as a sequence of UTF-16 / UCS-2 code units (the underlying string representation of the host language at `punycode.js:98`).
- Output: an ordered sequence (array) of non-negative integer code points, one entry per decoded code point (`punycode.js:102`, `punycode.js:122`).
- The input is not mutated; the routine only reads code units via positional indexing (`punycode.js:106`, `punycode.js:109`).
- No errors are raised for any input, whether it contains well-formed surrogate pairs, lone (unmatched) high surrogates, lone low surrogates, or arbitrary code-unit sequences. The body of `ucs2decode` contains no error-raising branch (`punycode.js:101-123`).
- Important clarification: the two RangeError assertions nested inside the `describe('punycode.ucs2.decode', ...)` block at `tests/tests.js:255-270` do NOT target `ucs2.decode`; they invoke `punycode.decode` (the Punycode-algorithm decoder). See the dedicated section below.

## Behavior rules
- A pair of code units where the first lies in the high-surrogate range `U+D800..U+DBFF` and the immediately following code unit lies in the low-surrogate range `U+DC00..U+DFFF` is decoded into a single supplementary-plane code point via the standard surrogate-pair combination formula `((high & 0x3FF) << 10) + (low & 0x3FF) + 0x10000`. Consecutive valid surrogate pairs each decode to one integer code point. Fixture: `tests/tests.js:141-144` (four astral code points `127829, 119808, 119558, 119638` from eight code units). Implementation: `punycode.js:107` (high-surrogate range check), `punycode.js:109-111` (low-surrogate detection and combination).
- When a high surrogate is encountered and the next code unit is NOT a low surrogate, the high surrogate is emitted as a single integer equal to its raw code-unit value (e.g. `0xD800 = 55296`) and the look-ahead cursor is rewound so the next code unit is re-examined from the start of the main loop. Fixture: `tests/tests.js:145-149` (encoded `\uD800ab` decodes to `[55296, 97, 98]`). Implementation: `punycode.js:112-117` (unmatched-surrogate branch, `output.push(value)` then `counter--`).
- When a low surrogate appears without being preceded by a high surrogate, it is emitted as a single integer equal to its raw code-unit value (e.g. `0xDC00 = 56320`). Low surrogates do not trigger the look-ahead path because that path is entered only on a high-surrogate value (`punycode.js:107`); therefore a low surrogate falls through to the default emission at `punycode.js:118-120`. Fixture: `tests/tests.js:150-154` (encoded `\uDC00ab` decodes to `[56320, 97, 98]`).
- When a high surrogate is immediately followed by another high surrogate, the first high surrogate is treated as unmatched (its look-ahead fails the low-surrogate test at `punycode.js:110`) and is emitted raw; the cursor is rewound at `punycode.js:116` so the second high surrogate is then processed from the top of the loop, fails its own look-ahead (no following code unit), and is also emitted raw. Fixture: `tests/tests.js:155-159` (encoded `\uD800\uD800` decodes to `[0xD800, 0xD800]`).
- An unmatched high surrogate, followed by a valid high-low surrogate pair, followed by another unmatched high surrogate, yields three output integers in order: the first unmatched high surrogate, the combined supplementary-plane code point, then the trailing unmatched high surrogate. Fixture: `tests/tests.js:160-164` (encoded `\uD800\uD834\uDF06\uD800` decodes to `[0xD800, 0x1D306, 0xD800]`). Implementation interaction: `punycode.js:110-117` (unmatched branch rewinds cursor so the next pair is re-examined; valid pair then combined at `punycode.js:111`).
- Two consecutive low surrogates are each emitted raw as separate integers, because low surrogates are never consumed by the look-ahead branch. Fixture: `tests/tests.js:165-169` (encoded `\uDC00\uDC00` decodes to `[0xDC00, 0xDC00]`). Implementation: `punycode.js:107` (high-surrogate gate) falls through to `punycode.js:118-120`.
- An unmatched low surrogate, followed by a valid high-low surrogate pair, followed by a trailing unmatched low surrogate, yields three integers: the leading lone low surrogate, the combined supplementary-plane code point, and the trailing lone low surrogate. Fixture: `tests/tests.js:170-174` (encoded `\uDC00\uD834\uDF06\uDC00` decodes to `[0xDC00, 0x1D306, 0xDC00]`).
- Additionally, the look-ahead path is only taken when a next code unit is available; when a high surrogate is the final code unit of the input, it is emitted raw because the bounds guard at `punycode.js:107` (`counter < length`) fails and execution falls to the default emission branch at `punycode.js:118-120`. This behavior is exercised implicitly by the trailing unmatched high surrogate in `tests/tests.js:160-164`.

## Cross-cutting RangeError tests in this describe block
Two `it` cases in the `describe('punycode.ucs2.decode', ...)` block assert that a `RangeError` is thrown, but both call the top-level Punycode decoder, not `ucs2.decode`:

- `tests/tests.js:255-262` invokes `punycode.decode('\x81-')`. The input has a trailing delimiter `-`, so the basic-code-point loop at `punycode.js:213-219` runs; the code unit `0x81` satisfies `>= 0x80` at `punycode.js:215`, which triggers the `error('not-basic')` call at `punycode.js:216`. The message text originates from `punycode.js:22-26` (the `errors` table; specifically `not-basic`: "Illegal input >= 0x80 (not a basic code point)").
- `tests/tests.js:263-270` invokes `punycode.decode('\x81')`. There is no delimiter, so `basic` becomes `0` and the main decoding loop at `punycode.js:224` starts at index 0. `basicToDigit(0x81)` returns a value `>= base`, exercising the overflow check path: the loop then calls `error('overflow')` at `punycode.js:243-244` when `digit > floor((maxInt - i) / w)` cannot hold. The message text originates from `punycode.js:22-26` (the `errors` table; specifically `overflow`: "Overflow: input needs wider integers to process").

A Go port MUST implement both error paths inside the Punycode decoder (the counterpart to `punycode.js:196-281`), and MUST ensure that the UCS-2 decoder (the counterpart to `punycode.js:101-123`) never raises an error on any input.

## Port notes (Go)
- The Go string type is a sequence of bytes typically holding UTF-8, not UTF-16. A faithful port cannot simply iterate over runes, because ranging over a Go string yields decoded runes (with replacement characters substituted for invalid UTF-8), which destroys the ability to observe unmatched surrogate halves as distinct integers. The source JSDoc at `punycode.js:90-94` makes the UCS-2-vs-UTF-16 distinction explicit.
- The Go port should expose an input surface that preserves UTF-16 code units as-is. Acceptable approaches include: accepting `[]uint16` directly; accepting a Go string whose contents are pre-arranged as UTF-16 code units (for example, produced by the caller from the original source representation); or providing an internal UTF-8-to-UTF-16 conversion that preserves lone surrogate halves when the caller supplies them deliberately. The decoding algorithm itself — iterate by index, test for high-surrogate range, peek and test for low-surrogate range, combine or emit raw with a rewind — is a direct translation of `punycode.js:101-123`.
- Output should be `[]rune` (or `[]int32` / `[]uint32`) since code-point values can exceed `0xFFFF` for supplementary-plane results and may equal raw unmatched surrogate values in `0xD800..0xDFFF`, which are not valid Unicode scalar values — so the Go port must NOT validate output integers against the scalar-value range. See `TARGET.md` for the overall Go port requirements.
